## AI 협업의 성공 요인

### 1. 명확한 단계 구분 (SPEC → FINAL)

**효과:**
- 각 단계의 역할이 명확하여 작업 범위 선정이 용이
- 의사결정이 단순화됨 (현재 단계 내에서만 판단)
- 주의 산만 최소화

**예시:**
- SPEC: 분석 → 지침 설정
- RED: 테스트 작성만 (구현 금지)
- GREEN: 최소 구현 (리팩토링 금지)
- REFACTOR: 기능 변화 금지
- FEATURE: 신규 기능만
- FINAL: 검증 및 정리

### 2. 상세한 요구사항 문서

**파일 구조:**
```
TODO.md              → 세부 체크리스트
AGENTS.md            → 프로젝트 지침
prompts/INDEX.md     → 프롬프트 매핑
prompts/phase-*/     → 단계별 상세 프롬프트
```

**효과:**
- 모호함 제거
- 구현 범위 명확화
- 자동 검증 기준 명시

### 3. 테스트 중심 접근 (TDD)

**흐름:**
1. RED: 요구사항을 테스트로 명시
2. GREEN: 테스트를 통과하는 최소 구현
3. 회귀 테스트: 기존 기능 유지 확인

**효과:**
- 요구사항 명확화
- 자동 검증 (50/50 통과)
- 동작 보증

### 4. 점진적 리팩토링

**특징:**
- 각 항목이 작음 (1~2개 함수)
- 단계적 검증 (매 항목마다)
- 위험도 낮음 (15개 REFACTOR 항목 모두 통과)

**효과:**
- 빠른 피드백
- 오류 신속 발견
- 신뢰 구축

## AI 협업의 한계점

### 1. 설계 변경의 어려움

**상황:**
- 초기 설계에서 기본 가정을 바꾸려면 비용 큼
- 예: Session API의 후속 개선

**영향:**
- 불가피한 재작업 발생 가능
- 초기 설계 검토의 중요성

**교훈:**
- 설계 단계 투자 중요
- 프로토타입 검증 추천

### 2. 문맥 길이 제약

**현상:**
- 긴 변경 이력 추적 어려움
- 이전 의사결정 맥락 손실 가능

**대응:**
- 문서화 필수
- SESSION_NOTES.md 활용
- 단계별 요약 유지

### 3. 자동화 한계

**제약:**
- 수동 검증 필요
- IDE 지원 부족 (CLI 환경)
- 자동화 불가능한 영역 존재

**예시:**
- CSV 파일 형식 검증
- 한글 텍스트 처리
- UI 동작 확인
