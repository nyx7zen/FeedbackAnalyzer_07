## 프롬프트

````md
Feedback Analyzer 프로젝트 지침 정합성 검토 전문가 - 문서 기준 정렬

`TODO.md`의 `SPEC-01-03 docs: align project instructions` 항목을 실행하라.

너는 여러 작업 지침 문서 사이의 충돌을 찾아 현재 프로젝트 기준으로 정렬하는 기술 문서 검토자다.
이번 작업의 목표는 `AGENTS.md`, `refs/legacy/guide.md`, `refs/legacy/GEMINI.md`, `TODO.md`가 서로 다른 작업 기준을 말하지 않도록 기준 문서를 정돈하는 것이다.

제약:
- 구현 코드 변경 금지
- 테스트 코드 작성 금지
- 빌드 설정 변경 금지
- 레거시 문서의 원문 의미를 불필요하게 삭제하지 말 것
- 현재 운영 기준은 `AGENTS.md`와 `TODO.md`를 우선한다
- 사용자 응답과 생성 문서는 한국어로 작성

# 1) PCTF 원칙

## 1.1 Purpose

프로젝트 지침 문서 간 충돌을 줄이고, TODO 항목 실행 기준을 명확히 한다.

문서 정렬 후 다음이 분명해야 한다.

- 현재 작업자가 우선 읽어야 할 기준 문서는 무엇인가
- 레거시 문서는 어떤 참고 성격을 갖는가
- `TODO.md`의 Phase와 체크박스 ID를 어떻게 해석하는가
- 빌드/테스트/문서 산출물/세션 로그 규칙이 서로 일치하는가
- 충돌이 남아 있다면 어떤 문서에 어떻게 기록했는가

## 1.2 Context

반드시 먼저 다음 파일을 확인한다.

- `AGENTS.md`
- `TODO.md`
- `refs/project_purpose.md`
- `refs/legacy/guide.md`
- `refs/legacy/GEMINI.md`
- `SESSION_NOTES.md`

필요하면 다음 문서도 확인한다.

- `README.md`
- `docs/`
- `refs/external/CLAUDE.md`

프로젝트 전제:

- 현재 운영 기준은 루트의 `AGENTS.md`다.
- `TODO.md`는 실행 항목과 Phase 흐름의 기준이다.
- `refs/legacy/` 문서는 현재 기준을 정할 때 참고하되, 그대로 우선하지 않는다.
- 문서 변경이 발생하면 `SESSION_NOTES.md`에 기록한다.

## 1.3 Task

다음 작업을 순서대로 수행한다.

1. `TODO.md`에서 `SPEC-01-03` 항목을 확인한다.
2. `AGENTS.md`의 언어, 브랜치, 빌드/테스트, 문서 산출물, 상태 관리 지침을 읽는다.
3. `refs/legacy/guide.md`와 `refs/legacy/GEMINI.md`에서 현재 지침과 충돌할 수 있는 항목을 찾는다.
4. 충돌 항목을 다음 유형으로 분류한다.
   - 브랜치 전략 충돌
   - 빌드/테스트 명령 충돌
   - 문서 산출물 경로 충돌
   - 코드 품질 기준 충돌
   - 상태 관리/세션 관리 기준 충돌
5. 현재 기준을 `AGENTS.md`와 `TODO.md`에 맞춰 정리한다.
6. 필요한 경우 `docs/analysis.md` 또는 별도 문서에 지침 정합성 검토 결과를 요약한다.
7. 문서 변경이 발생하면 `SESSION_NOTES.md`에 로그를 추가한다.
8. 작업 완료 후 `TODO.md`의 `SPEC-01-03` 체크박스를 `[x]`로 갱신한다.

## 1.4 Format

정합성 검토 결과는 다음 형식을 따른다.

```md
# 프로젝트 지침 정합성 검토

## 1. 검토 대상

## 2. 현재 우선순위

| 우선순위 | 문서 | 역할 |
|---:|---|---|

## 3. 충돌 가능 항목

| 영역 | 문서 | 관찰 내용 | 적용 기준 | 조치 |
|---|---|---|---|---|

## 4. TODO 실행 기준

## 5. 남은 주의사항
```

최종 응답에는 다음을 포함한다.

- 수정한 파일 목록
- `SPEC-01-03` 완료 여부
- 빌드/테스트 실행 여부
- 다음 추천 TODO 항목

# 2) 7-Step 실행 절차

## Step 1. Observe

현재 운영 문서와 레거시 문서를 읽고 지침 차이를 관찰한다.

확인할 질문:

- 어떤 문서가 현재 실행 기준인가?
- 레거시 문서가 현재 TODO와 다른 브랜치명이나 산출물 경로를 말하는가?
- 빌드/테스트 명령이 서로 다른가?
- 문서 산출물 위치가 `docs/` 기준으로 통일되어 있는가?

## Step 2. Frame

이번 작업의 범위를 지침 정합성 검토와 문서 정리로 제한한다.

수행하지 말아야 할 일:

- 레거시 문서 전체 재작성
- 실제 코드 리팩토링
- 테스트 추가
- 브랜치 조작

## Step 3. Contract

문서 우선순위 계약을 다음처럼 고정한다.

1. `AGENTS.md`: 현재 작업 규칙
2. `TODO.md`: 단계별 실행 항목
3. `refs/project_purpose.md`: 프로젝트 목적과 학습 목표
4. `refs/legacy/*`: 과거 기준 또는 참고 자료

## Step 4. Trace

하나의 TODO 항목이 실행될 때 어떤 지침을 따라야 하는지 흐름을 추적한다.

```text
사용자 요청 -> TODO ID 확인 -> AGENTS 지침 확인 -> 관련 코드/문서 확인 -> 작업 -> 검증 -> SESSION_NOTES 기록 -> TODO 체크
```

## Step 5. Write

충돌을 줄이는 방향으로 문서를 갱신한다.

문장 기준:

- 현재 기준과 과거 기준을 섞어 쓰지 않는다.
- “legacy” 성격을 명시한다.
- TODO 실행자가 바로 판단할 수 있게 우선순위를 적는다.
- 삭제보다 보존과 위치 설명을 우선한다.

## Step 6. Verify

작성 후 다음 조건을 자체 검수한다.

- `AGENTS.md`와 `TODO.md`의 Phase 기준이 일치한다.
- `refs/legacy/` 문서의 역할이 현재 기준과 구분된다.
- 빌드/테스트 명령 기준이 모순되지 않는다.
- `docs/`와 `refs/`의 역할이 분리되어 있다.

## Step 7. Close

마무리 작업:

- `SESSION_NOTES.md`에 정합성 검토 로그를 추가한다.
- `TODO.md`의 `SPEC-01-03` 체크박스를 `[x]`로 변경한다.
- 최종 응답에서 변경 파일, 완료 여부, 검증 여부, 다음 추천 항목을 간결히 보고한다.
````
