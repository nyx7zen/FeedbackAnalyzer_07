# 1) PCTF 원칙

## 1.1 Purpose

현재 Feedback Analyzer 구조를 분석해 이후 RED, GREEN, REFACTOR, FEATURE 단계의 기준 문서를 만든다.

분석 문서에는 다음 목적이 드러나야 한다.

- 라우팅, UI 렌더링, 도메인 분석, 필터링, 세션 상태가 어떻게 연결되는지
- 어떤 구조가 테스트 작성을 어렵게 만드는지
- 어떤 리팩토링이 가장 먼저 필요한지
- 기존 동작을 보호하기 위해 어떤 테스트가 필요한지
- TODO 단계가 어떤 구조 개선 흐름으로 이어지는지

## 1.2 Context

반드시 먼저 다음 파일을 확인한다.

- `TODO.md`
- `AGENTS.md`
- `refs/project_purpose.md`
- `src/main.cpp`
- `src/TextAnalyzer.cpp`
- `src/TextAnalyzer.h`
- `src/Filters.cpp`
- `src/Filters.h`
- `src/Session.cpp`
- `src/Session.h`
- `src/Constants.cpp`
- `src/Constants.h`
- `src/FileHandler.cpp`
- `src/FileHandler.h`

파일이 없거나 경로가 다르면 저장소의 실제 구조를 우선한다.

프로젝트 전제:

- `main.cpp`의 라우팅/UI 렌더링과 도메인 로직 결합을 줄이는 방향을 우선한다.
- 전역 `static` 상태는 테스트 격리 위험으로 본다.
- 중복 문자열 검색 로직은 공용 유틸리티 후보로 본다.
- 축약 함수명과 하드코딩 문자열은 리팩토링 후보로 본다.

## 1.3 Task

다음 작업을 순서대로 수행한다.

1. `TODO.md`에서 `SPEC-02-01` 항목을 확인한다.
2. 주요 소스 파일을 읽고 모듈별 책임을 식별한다.
3. 피드백 입력부터 분석, 필터링, 출력, 저장까지의 흐름을 추적한다.
4. 구조적 문제를 다음 범주로 분류한다.
   - 책임 집중
   - UI/도메인 결합
   - 전역 상태
   - 테스트 격리 위험
   - 중복 로직
   - 네이밍/상수화 문제

|---|---|---|---|

|---|---:|---|---|

- 빌드/테스트 실행 여부
- 다음 추천 TODO 항목

# 2) 7-Step 실행 절차

## Step 1. Observe

소스 파일과 문서를 읽고 현재 구조를 관찰한다.

확인할 질문:

- HTTP 라우팅은 어디에 있는가?
- HTML 렌더링은 어디에서 수행되는가?
- 감성 분석과 키워드 분석은 어떤 함수가 담당하는가?
- 필터링 상태는 어디에 저장되는가?
- 세션 API는 실제 상태 저장소인가?

## Step 2. Frame

이번 작업은 분석 문서 작성이다.

수행하지 말아야 할 일:

- 소스 코드 수정
- 테스트 작성
- CMake 변경
- 리팩토링 실행

## Step 3. Contract

분석 기준을 다음 계약으로 고정한다.

- 실제 파일명과 함수명에 근거한다.
- 위험은 테스트로 보호 가능한 형태로 표현한다.
- 개선 방향은 TODO 항목과 연결한다.
- 아직 확인하지 못한 내용은 단정하지 않는다.

## Step 4. Trace

요청 처리 흐름을 추적한다.

예상 흐름:

```text
HTTP 요청 -> 입력 파싱 -> 텍스트 분석 -> 필터 적용 -> HTML 렌더링 -> 저장 또는 응답
```

실제 코드가 다르면 실제 흐름을 우선 기록한다.

## Step 5. Write

`docs/analysis.md`를 작성한다.

문장 기준:

- 파일명과 함수명을 포함한다.
- “좋지 않음” 대신 어떤 테스트/리팩토링 위험인지 쓴다.
- 개선 방향은 `RED`, `GREEN`, `REFACTOR`, `FEATURE`와 연결한다.

## Step 6. Verify

작성 후 다음 조건을 자체 검수한다.

- `docs/analysis.md`가 존재한다.
- 실제 코드 구조와 충돌하지 않는다.
- 주요 모듈 책임 표가 포함되어 있다.
- 리팩토링 위험도 표가 포함되어 있다.
- TODO 후속 단계와 연결되어 있다.
- 문서 작업만 수행했다면 빌드/테스트 미실행을 기록한다.

## Step 7. Close

마무리 작업:

- `SESSION_NOTES.md`에 분석 문서 작성 로그를 추가한다.
- `TODO.md`의 `SPEC-02-01` 체크박스를 `[x]`로 변경한다.
- 최종 응답에서 변경 파일, 완료 여부, 검증 여부, 다음 추천 항목을 간결히 보고한다.
