# 1) PCTF 원칙

## 1.1 Purpose

SPEC 단계 분석 결과를 바탕으로 후속 REFACTOR 단계의 작업 순서를 명확히 한다.

문서에는 다음 목적이 드러나야 한다.

- 어떤 리팩토링이 가장 먼저 필요한지
- 테스트 보호 없이 건드리면 위험한 영역은 무엇인지
- 네이밍, 상수화, 중복 제거, 상태 관리 중 어떤 순서가 안전한지
- RED/GREEN 단계에서 먼저 보호해야 할 동작은 무엇인지
- TODO 항목 간 의존관계가 어떻게 이어지는지

## 1.2 Context

반드시 먼저 다음 파일을 확인한다.

- `TODO.md`
- `AGENTS.md`
- `docs/analysis.md`
- `src/main.cpp`
- `src/TextAnalyzer.cpp`
- `src/TextAnalyzer.h`
- `src/Filters.cpp`
- `src/Filters.h`
- `src/Session.cpp`
- `src/Session.h`
- `src/Constants.cpp`
- `src/Constants.h`

파일이 없거나 경로가 다르면 저장소의 실제 구조를 우선한다.

우선순위 평가 기준:

- 사용자 동작 영향도
- 테스트 부재 위험
- 변경 범위
- 전역 상태 의존도
- 중복 정도
- 후속 TODO와의 연결성

## 1.3 Task

다음 작업을 순서대로 수행한다.

1. `TODO.md`에서 `SPEC-02-04` 항목을 확인한다.
2. `docs/analysis.md`의 기존 분석 내용을 확인한다.
3. 네이밍 스멜 후보를 찾는다.
   - `sent`
   - `kw`
   - `fil`
   - 의미가 불분명한 축약 이름
4. 하드코딩 문자열과 매직 넘버 후보를 찾는다.
5. 중복 문자열 검색 로직 후보를 찾는다.
6. 전역 상태와 세션 상태 관리 문제를 우선순위에 포함한다.
7. `docs/analysis.md`에 리팩토링 우선순위 섹션을 추가하거나 갱신한다.
8. 문서 변경이 발생하면 `SESSION_NOTES.md`에 로그를 추가한다.
9. 작업 완료 후 `TODO.md`의 `SPEC-02-04` 체크박스를 `[x]`로 갱신한다.

## 1.4 Format

`docs/analysis.md`에 다음 섹션을 추가한다.

```md
## 리팩토링 우선순위

| 순위 | 대상 | 문제 유형 | 근거 | 선행 보호 | 연결 TODO |
|---:|---|---|---|---|---|

### 1차 보호 대상

### 2차 구조 개선 대상

### 3차 기능 확장 전 정리 대상

### 권장 진행 순서
```

권장 진행 순서는 다음 TODO 흐름과 연결한다.

```text
SPEC 분석 -> RED 실패 테스트 -> GREEN 최소 수정 -> REFACTOR 구조 개선 -> FEATURE 기능 확장 -> final 검수
```

리팩토링 후보와 TODO 연결 예:

| 후보 | 연결 TODO |
|---|---|
| 축약 함수명 | `REFACTOR-01-01` ~ `REFACTOR-01-03` |
| 감성 라벨 하드코딩 | `REFACTOR-01-04` |
| 점수 매직 넘버 | `REFACTOR-01-05` |
| 중복 키워드 | `REFACTOR-01-06` |
| 중복 문자열 검색 | `REFACTOR-02-01` ~ `REFACTOR-02-03` |
| 긴 분석/필터 함수 | `REFACTOR-02-04`, `REFACTOR-02-05` |
| 전역 상태 | `REFACTOR-03-01`, `REFACTOR-03-02` |
| 세션 저장소 | `REFACTOR-03-03`, `REFACTOR-03-04`, `REFACTOR-03-05` |

최종 응답에는 다음을 포함한다.

- 수정한 파일 목록
- `SPEC-02-04` 완료 여부
- 빌드/테스트 실행 여부
- 다음 추천 TODO 항목

# 2) 7-Step 실행 절차

## Step 1. Observe

분석 문서와 코드를 읽고 리팩토링 후보를 관찰한다.

권장 검색:

```powershell
rg -n "sent\\(|kw\\(|fil\\(|긍정|부정|중립|0\\.0f|1\\.0f|static|find\\(" src
```

확인할 질문:

- 이름만 보고 의미가 불분명한 함수가 있는가?
- 감성 라벨 문자열이 여러 곳에 반복되는가?
- 문자열 검색 로직이 복제되어 있는가?
- 상태 관리 문제가 테스트를 방해하는가?
- 기능 추가 전에 정리해야 할 위험이 있는가?

## Step 2. Frame

이번 작업은 우선순위 문서화다.

수행하지 말아야 할 일:

- 함수명 변경
- 상수 추출
- 유틸리티 생성
- 전역 상태 제거
- 테스트 작성

## Step 3. Contract

우선순위 판단 기준을 다음처럼 고정한다.

- 테스트로 먼저 보호해야 하는 동작은 RED/GREEN보다 앞서 리팩토링하지 않는다.
- 변경 범위가 작고 의미가 명확한 네이밍 개선은 초반 후보로 둔다.
- 전역 상태 제거는 테스트 보호와 초기화 경로 확보 후 진행한다.
- 기능 확장은 구조 개선 뒤에 진행한다.

## Step 4. Trace

후속 TODO 의존 흐름을 추적한다.

```text
문제 발견 -> 실패 테스트 작성 -> 최소 수정 -> 이름/상수/중복 정리 -> 상태 캡슐화 -> 기능 확장
```

## Step 5. Write

`docs/analysis.md`를 갱신한다.

문장 기준:

- 순위마다 근거와 선행 보호를 함께 쓴다.
- TODO ID를 명시한다.
- “나중에 정리” 대신 어느 Phase에서 다룰지 적는다.
- 대규모 재설계 대신 현재 TODO 범위의 개선 순서를 제시한다.

## Step 6. Verify

작성 후 다음 조건을 자체 검수한다.

- 우선순위 표가 포함되어 있다.
- 네이밍, 중복 코드, 상태 관리 문제가 모두 포함되어 있다.
- 각 항목이 TODO ID와 연결되어 있다.
- 테스트 보호가 필요한 항목이 구분되어 있다.
- 문서 작업만 수행했다면 빌드/테스트 미실행을 기록한다.

## Step 7. Close

마무리 작업:

- `SESSION_NOTES.md`에 리팩토링 우선순위 문서화 로그를 추가한다.
- `TODO.md`의 `SPEC-02-04` 체크박스를 `[x]`로 변경한다.
- 최종 응답에서 변경 파일, 완료 여부, 검증 여부, 다음 추천 항목을 간결히 보고한다.
